# 第9章 收藏同步与跨设备数据

> Kazumi 支持 WebDAV 和 Bangumi 双通道跨设备同步。
> 收藏同步用 Change Log 模式，历史同步用 Event Sourcing 模式。
> 两种模式解决同一个问题：多台设备间的追番数据保持一致。

---

## 9.3 收藏同步 — Change Log 模式

### 数据模型

```dart
// 收藏项
class CollectedBangumi {
    BangumiItem bangumiItem;  // 番剧信息
    DateTime time;            // 收藏时间
    int type;                 // 收藏类型：1=在看, 2=想看, 3=搁置, 4=看过, 5=抛弃
}

// 变更日志
class CollectedBangumiChange {
    int id;          // 时间戳作为唯一ID
    int bangumiID;   // 番剧ID
    int action;      // 操作：1=添加, 2=更新, 3=删除
    int type;        // 收藏类型
    int timestamp;   // 时间戳
}
```

### 用户操作 → 记录变更

```dart
// CollectController 中
Future<void> addCollect(BangumiItem bangumiItem, int type) async {
    // 1. 写入收藏
    await collectRepository.addCollect(bangumiItem, type);
    // 2. 记录变更日志
    await GStorage.appendCollectChange(CollectedBangumiChange(
        id: DateTime.now().millisecondsSinceEpoch,
        bangumiID: bangumiItem.id,
        action: 1,  // 添加
        type: type,
        timestamp: DateTime.now().millisecondsSinceEpoch,
    ));
}
```

### WebDAV 合并算法

> 源码：`lib/modules/collect/collect_sync_merger.dart` → `mergeWebDav()`

```dart
static CollectiblesMergeResult mergeWebDav({
    required List<CollectedBangumi> localCollectibles,
    required List<CollectedBangumiChange> localChanges,
    required List<CollectedBangumi> remoteCollectibles,
    required List<CollectedBangumiChange> remoteChanges,
}) {
    // 1. 以远程收藏为基础
    final mergedCollectibles = remoteCollectibles.map(_copyCollectible).toList();

    // 2. 找出本地新增的 change（通过 ID 去重）
    final newLocalChanges = localChanges.where((localChange) {
        return !remoteChanges.any((r) => r.id == localChange.id);
    }).toList()
      ..sort((a, b) => a.timestamp.compareTo(b.timestamp));

    // 3. 按时间顺序回放本地 change
    for (final change in newLocalChanges) {
        if (change.action == 3) {
            // 删除
            mergedCollectibles.removeWhere((item) => item.bangumiItem.id == change.bangumiID);
        } else if (change.action == 1) {
            // 添加（有则替换，无则追加）
            mergedCollectibles.add(changedCollectible);
        } else if (change.action == 2) {
            // 更新
            mergedCollectibles[index] = changedCollectible;
        }
    }

    // 4. 合并变更日志（去重）
    final mergedChanges = <int, CollectedBangumiChange>{
        for (final change in remoteChanges) change.id: change,
        for (final change in newLocalChanges) change.id: change,
    }.values.toList();

    return CollectiblesMergeResult(collectibles: mergedCollectibles, changes: mergedChanges);
}
```

**核心思想：远程为基础 + 回放本地新增 change = 合并结果。**

### Bangumi 同步算法

> 源码：`lib/modules/collect/collect_sync_merger.dart` → `planBangumi()`

```dart
static BangumiCollectiblesMergePlan planBangumi({
    required List<CollectedBangumi> localCollectibles,
    required List<BangumiCollection> remoteCollections,
    required BangumiSyncPriority priority,  // localFirst 或 bangumiFirst
}) {
    // 1. 计算集合差异
    localOnlyIds  = 本地有，远程没有   → 上传到 Bangumi
    remoteOnlyIds = 远程有，本地没有   → 写入本地
    mismatchIds   = 都有但类型不同    → 按优先级解决冲突

    // 2. 冲突解决
    if (priority == localFirst) {
        // 以本地为准 → 上传到 Bangumi
        conflictUploads.add(BangumiUploadMutation(id, localType));
    } else {
        // 以 Bangumi 为准 → 更新本地
        conflictLocalUpdates.add(BangumiLocalMutation(remoteCollectible));
    }

    // 3. 返回四阶段执行计划
    return BangumiCollectiblesMergePlan(
        localOnlyUploads,    // 阶段1：本地独有的上传
        remoteOnlyPuts,      // 阶段2：远程独有的写入本地
        conflictUploads,     // 阶段3：冲突（本地优先时上传）
        conflictLocalUpdates,// 阶段4：冲突（远程优先时更新本地）
    );
}
```

## 9.4 历史同步 — Event Sourcing 模式

### 什么是事件溯源

传统方式直接存储最终状态：

```
传统：History { 看到第3集, 进度125000ms }  ← 只有最终状态，不知道怎么来的
```

事件溯源不存最终状态，而是记录所有变更事件：

```
Event Sourcing：
  Event 1: { op: upsertProgress, episode: 1, progressMs: 60000 }
  Event 2: { op: upsertProgress, episode: 1, progressMs: 120000 }
  Event 3: { op: upsertProgress, episode: 2, progressMs: 30000 }
  Event 4: { op: upsertProgress, episode: 3, progressMs: 125000 }
    ↓ 回放所有事件
History { 看到第3集, 进度125000ms }  ← 最终状态是回放出来的
```

**优势：多端合并天然支持。每台设备记录自己的事件，同步时把所有设备的事件合在一起回放就行。**

### 数据模型

> 源码：`lib/modules/history/history_sync.dart`

```dart
// 操作类型
enum HistorySyncOp {
    upsertProgress,  // 更新观看进度
    deleteHistory,   // 删除历史记录
    clearAll,        // 清空所有历史
}

// 同步事件（不可变）
class HistorySyncEvent {
    String eventId;      // "设备UUID:序列号"，如 "550e8400-...:42"
    String deviceId;     // 设备标识
    int seq;             // 设备内序列号（递增）
    HistorySyncOp op;    // 操作类型
    int updatedAt;       // 时间戳（毫秒）
    String? entityKey;   // 番剧标识（如 "AGE12345"）
    BangumiItem? bangumiItem;  // 番剧信息
    String? adapterName; // 规则名
    int? episode;        // 集数
    int? road;           // 播放源
    int? progressMs;     // 播放位置（毫秒）
    String? lastSrc;     // 最后播放地址
    String? lastWatchEpisodeName; // 最后观看集名
}

// 快照（某一时点的全量数据）
class HistorySyncSnapshot {
    int generatedAt;                          // 快照生成时间
    List<History> histories;                  // 所有历史记录
    Map<String, String> itemVersions;         // 每个番剧的版本号
    Map<String, Map<int, String>> progressVersions;  // 每集的版本号
    Map<String, String> deletedVersions;      // 删除操作的版本号
    String? clearVersion;                     // 清空操作的版本号
}

// 版本号（字典序比较，确定性排序）
class HistorySyncVersion {
    static String of({required int updatedAt, required String eventId}) {
        return '${updatedAt.toString().padLeft(16, '0')}|$eventId';
        // 例如："000001718000000000|550e8400-...:42"
    }
    static int compare(String a, String b) => a.compareTo(b);
}
```

### 版本号比较规则

```
版本号格式："000001718000000000|device-uuid:42"
            ─────────────────  ──────────────
            时间戳（16位补零）   事件ID

比较规则（字典序）：
  1. 先比时间戳 → 时间戳大的胜出
  2. 时间戳相同 → 比事件ID → 字典序大的胜出

示例：
  "000001718000000000|device-A:1" < "000001718000000000|device-B:1"
  "000001718000000000|device-A:1" < "000001719000000000|device-A:1"
```

**确定性排序，不需要人工干预，任何两台设备用相同输入都会得到相同结果。**

### 合并引擎

> 源码：`lib/modules/history/history_sync.dart` → `HistorySyncMerger`

```dart
class HistorySyncMerger {
    static HistorySyncSnapshot merge({
        required HistorySyncSnapshot snapshot,  // 基础快照
        required Iterable<HistorySyncEvent> events,  // 所有设备的事件
    }) {
        // 1. 从快照构建可变状态
        final state = HistorySyncState.fromSnapshot(snapshot);

        // 2. 按版本号排序所有事件
        final sortedEvents = events.toList()
            ..sort((a, b) => HistorySyncVersion.compare(a.version, b.version));

        // 3. 逐个回放事件
        for (final event in sortedEvents) {
            state.apply(event);
        }

        // 4. 输出新快照
        return state.toSnapshot();
    }
}
```

### 状态机：apply(event)

```dart
void apply(HistorySyncEvent event) {
    switch (event.op) {
        case upsertProgress: _applyUpsert(event);
        case deleteHistory:  _applyDelete(event);
        case clearAll:       _applyClear(event);
    }
}

void _applyUpsert(HistorySyncEvent event) {
    // 1. 检查是否比清空操作更新
    if (!_isNewerThanClear(event.version)) return;

    // 2. 检查是否比删除操作更新
    if (deletedVersion != null && event.version <= deletedVersion) return;

    // 3. 更新番剧信息（如果版本更新）
    if (event.version >= itemVersions[entityKey]) {
        history.bangumiItem = bangumiItem;
        history.lastWatchEpisode = episode;
        itemVersions[entityKey] = event.version;
    }

    // 4. 更新观看进度（如果版本更新）
    if (event.version >= progressVersions[entityKey][episode]) {
        history.progresses[episode] = Progress(episode, road, progressMs);
        progressVersions[entityKey][episode] = event.version;
    }
}
```

**关键：每次更新都检查版本号，只有更新的事件才会生效。这就是冲突解决的机制。**

### 事件日志服务

> 源码：`lib/services/sync/history_sync_service.dart`

```dart
class HistorySyncService {
    // 单例
    static final HistorySyncService _instance = HistorySyncService._internal();
    factory HistorySyncService() => _instance;

    // 设备ID（UUID v4，首次生成后持久化）
    Future<String> getDeviceId() async {
        final existing = GStorage.getSetting(SettingsKeys.historySyncDeviceId);
        if (existing.isNotEmpty) return existing;
        final deviceId = HistorySyncDevice.generateDeviceId();
        await GStorage.putSetting(SettingsKeys.historySyncDeviceId, deviceId);
        return deviceId;
    }

    // 写入事件到本地 JSONL 文件（追加模式）
    Future<void> appendEvent(HistorySyncEvent event) async {
        final file = await localChangeLogFile();
        await file.writeAsString(
            HistorySyncCodec.eventsToJsonLines([event]),
            mode: FileMode.append,  // 追加，不覆盖
        );
    }

    // 压缩检查（日志 > 1MB 时需要压缩）
    Future<bool> shouldCompactLocalLog() async {
        final file = await localChangeLogFile();
        return file.lengthSync() > 1024 * 1024;
    }
}
```

### 事件发射点

> 源码：`lib/repositories/history_repository.dart`

所有历史写操作都会自动发射同步事件：

```dart
class HistoryRepository implements IHistoryRepository {
    Future<void> updateHistory(History history, int episode, int road, int progressMs) async {
        // 1. 更新本地 Hive
        await GStorage.histories.put(history.key, history);
        // 2. 发射同步事件
        await HistorySyncService().appendSafely(() =>
            HistorySyncService().appendUpsertProgress(
                history: history, episode: episode, road: road, progressMs: progressMs,
            ));
    }

    Future<void> deleteHistory(History history) async {
        await GStorage.histories.delete(history.key);
        await HistorySyncService().appendSafely(() =>
            HistorySyncService().appendDeleteHistory(history: history));
    }

    Future<void> clearAllHistories() async {
        await GStorage.histories.clear();
        await HistorySyncService().appendSafely(() =>
            HistorySyncService().appendClearAll());
    }
}
```

## 9.5 WebDAV 同步完整流程

> 源码：`lib/services/sync/webdav.dart` → `_syncHistory()`

```dart
Future<void> _syncHistory() async {
    final historySync = HistorySyncService();

    // 1. 读取本地事件
    final localEvents = await historySync.readLocalEvents();

    // 2. 从本地 Hive 构造合成事件（兼容首次同步）
    final localStateEvents = await _eventsFromLocalHistories();

    // 3. 读取远程快照
    var remoteSnapshot = await _readHistorySnapshot() ?? HistorySyncSnapshot.empty();

    // 4. 读取所有设备的远程事件
    final remoteEvents = await _readRemoteHistoryEvents();

    // 5. 如果远程没有任何数据，尝试导入旧版备份
    if (remoteSnapshot.histories.isEmpty && remoteEvents.isEmpty) {
        await _tryImportLegacyHistory();
    }

    // 6. ★ 合并：快照 + 所有事件 → 新快照
    final mergedSnapshot = HistorySyncMerger.merge(
        snapshot: remoteSnapshot,
        events: [...remoteEvents, ...localStateEvents, ...localEvents],
    );

    // 7. 将合并结果写入本地 Hive
    await historySync.applySnapshotToLocal(mergedSnapshot);

    // 8. 上传本地事件到远程
    if (localEvents.isNotEmpty) {
        await _writeDeviceHistoryChanges(localEvents);
    }

    // 9. 压缩（如果日志过大）
    if (await historySync.shouldCompactLocalLog()) {
        await _writeHistorySnapshot(mergedSnapshot);    // 写入新快照
        await historySync.replaceLocalEvents(const []);  // 清空本地事件
        await _writeDeviceHistoryChanges(const []);      // 清空远程设备日志
    }
}
```

### 压缩机制

```
压缩前：
  /history/snapshot.json          ← 旧快照
  /history/changes/device-A.jsonl ← 10000 条事件
  /history/changes/device-B.jsonl ← 8000 条事件
  本地 history.local.jsonl        ← 5000 条事件（> 1MB）

压缩后：
  /history/snapshot.json          ← 新快照（包含所有事件回放后的结果）
  /history/changes/device-A.jsonl ← 清空
  /history/changes/device-B.jsonl ← 清空
  本地 history.local.jsonl        ← 清空

效果：事件日志 → 压缩为快照，减少存储空间和同步时间
```

## 9.7 两种同步模式对比

| 对比项 | 收藏同步 (Change Log) | 历史同步 (Event Sourcing) |
|--------|----------------------|--------------------------|
| 记录方式 | 变更日志（增/删/改） | 不可变事件流 |
| 合并策略 | 远程为基础 + 回放本地 change | 快照 + 回放所有设备事件 |
| 冲突解决 | Bangumi 优先级策略 | 版本号字典序 |
| 存储格式 | Hive Box (二进制) | JSONL (文本行) |
| 压缩 | 无 | 日志 >1MB 时压缩为快照 |
| 多端支持 | WebDAV + Bangumi 双通道 | 仅 WebDAV |
| 实时同步 | Bangumi 支持单条实时推送 | 仅全量同步 |

## 9.9 总结

### 收藏同步链路

```
用户操作（添加/更新/删除收藏）
    ↓
CollectController → 记录 CollectedBangumiChange
    ↓
WebDAV: 下载远程 → mergeWebDav() → 回放本地 change → 上传
Bangumi: 拉取远程 → planBangumi() → 集合差异 + 优先级 → 四阶段执行
```

### 历史同步链路

```
用户操作（观看进度变化）
    ↓
HistoryRepository → HistorySyncService.appendEvent()
    ↓ 写入 history.local.jsonl
    ↓
WebDAV 同步：
  读取本地事件 + 读取远程快照 + 读取远程事件
    ↓
  HistorySyncMerger.merge(snapshot, allEvents)
    ↓ 按版本号排序 → 逐个回放 → 解决冲突
  应用到本地 Hive + 上传到远程
    ↓
  日志过大时压缩：事件 → 快照
```

### 核心设计思想

1. **Change Log**：记录"做了什么"，合并时回放。适合收藏这种简单 CRUD。
2. **Event Sourcing**：记录所有事件，不存最终状态。适合历史这种需要精确冲突解决的场景。
3. **版本号字典序**：`时间戳|eventId`，确定性排序，任何两台设备得到相同结果。
4. **快照 + 事件**：快照是基础，事件是增量。定期压缩事件为快照，控制存储大小。
5. **双通道**：WebDAV 全量同步 + Bangumi 单条推送，互补使用。
